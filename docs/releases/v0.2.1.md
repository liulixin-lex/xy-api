# v0.2.1

`v0.2.1` is the payment orchestration and checkout-quality release for the
protected `v0.2` line. It keeps the canonical quote, order, event, ledger,
idempotency, callback, credential-generation, and manual-review foundation
introduced in `v0.2.0`, while removing browser-bound provider creation and
separating public checkout contracts from administrator diagnostics.

This file describes the source release candidate. A `v0.2.1` tag starts the
protected staging workflow; that tag and any run-scoped candidate images are
not stable, deployable, or the final release alias until every blocking gate
passes and the publish-once workflow completes. The existing `v0.1.6` and
`v0.2.0` tags, images, assets, and release notes remain immutable.

## Product result

- Every supported online-payment route returns a local order immediately and
  continues on the site's own `/payment/:tradeNo` page while provider creation
  runs as durable background work. This includes Epay, Stripe, XORPay, Creem,
  Waffo, and Waffo Pancake without merging their gateway identities.
- The canonical public payment API uses opaque route IDs and public brands. It
  does not serialize the internal gateway, upstream order state, credential
  generation, webhook diagnostics, raw provider errors, or manual accounting
  terminology.
- Epay, Stripe, and XORPay remain independent gateway groups. Multiple routes
  for the same public brand can coexist without collapsing Epay Alipay/WeChat,
  XORPay Alipay/WeChat, and Stripe into one provider.
- Creem, Waffo, and Waffo Pancake remain independent integrations and are never
  presented as XORPay routes. Their compatibility endpoints now create the same
  canonical local orders and return only a first-party order reference; hosted
  instructions stay behind authenticated continuation and all three participate
  in the merchant-limit reservation engine.
- Default and Classic have separate canonical user checkout pages and
  administrator payment controls, while sharing only presentation-level
  primitives and stable backend contracts.

## Public payment route model

The browser selects an opaque `route_id`. Product- and option-based routes also
use opaque `product_id` or `option_id` values. The server resolves those public
identifiers to configured provider data. The same public method can therefore
have more than one independently configured route without exposing a gateway,
provider product ID, or upstream payment-option value.

| Public method | Gateway implementation | Capability |
| --- | --- | --- |
| Alipay | Epay | Existing configured Epay redirect/form flow |
| Alipay | XORPay | Alipay face-to-face QR; mobile may open only a strictly validated `qr.alipay.com` URL |
| WeChat Pay | Epay | Existing configured Epay redirect/form flow |
| WeChat Pay | XORPay | Native QR on desktop, JSAPI inside WeChat when explicitly enabled |
| Card | Stripe | One-time Checkout Session in `payment` mode |
| Online payment | Creem | Fixed configured product Checkout |
| Online payment | Waffo | Configured payment-option Checkout |
| Online payment | Waffo Pancake | Hosted Checkout |

The UI labels duplicate public brands as separate recommended or backup
channels without revealing the internal provider. XORPay remains visible only
in administrator and operations interfaces.

## Durable payment creation and recovery

Payment start now performs only local, transactional work in the browser
request:

1. Validate the server-authoritative quote and idempotency key.
2. Create or reuse the canonical local order.
3. Atomically reserve merchant-limit capacity when a policy applies.
4. Enqueue an idempotent provider-create task.
5. Return the local trade number and payment-page expiry.

Workers on every node claim tasks through database compare-and-swap leases.
Each claim advances a fencing token copied to the order. A worker whose lease
expired cannot commit a provider result over a newer worker.

The durable states distinguish pending creation, leased creation, ready
checkout instructions, awaiting payment, provider confirmation, fulfillment,
retry wait, ambiguous creation, permanent failure, expiry, and manual review.
User polling reads only the local database. Provider queries are performed by
background reconciliation and recovery tasks, preventing browser tabs from
amplifying upstream requests.

For an ambiguous XORPay create result, the worker queries before retrying:

- `not_exist`: require repeated confirmation before another create attempt.
- `new`: do not create a duplicate; move the order to manual review because
  XORPay cannot reconstruct the original QR or JSAPI instructions by query.
- `payed` or `success`: process the normalized paid event through the normal
  exactly-once settlement path.
- `expire`: expire the local order and release active capacity.
- `fee_error`: retain the order for administrator review.

The XORPay create request carries the remaining local lifetime. A provider
response cannot extend the site's authoritative order expiry.

Waffo uses a stable, order-derived payment request ID. Its recovery worker
queries that exact ID and requires repeated `not_exist` confirmation before it
may recreate the payment. Creem and Waffo Pancake do not expose an equivalent
reliable reconstruction contract for a lost hosted instruction, so an
ambiguous create result moves to manual review instead of blindly creating a
second provider order.

## Checkout behavior by environment

Stripe Checkout redirects remain server-side and are revalidated when created,
recovered, and opened through the authenticated continuation endpoint.
Stripe-owned `*.stripe.com` hosts are always allowed; optional custom Checkout
domains require an administrator-managed exact-host allowlist. Wildcards,
userinfo, ports, IP addresses, localhost, and non-DNS entries are rejected.
Existing entries cannot be removed while unfinished Stripe orders may still
depend on them.

### Alipay through XORPay

- Desktop shows the validated Alipay face-to-face QR inside the site's branded
  checkout page.
- Mobile offers an "Open Alipay" action only for a validated HTTPS
  `qr.alipay.com` destination.
- If app opening is unavailable, the page retains a QR or system-browser
  fallback.
- The product is not described as Alipay official PC website, WAP, or Alipay
  JSAPI access.

### WeChat Pay through XORPay

- Desktop uses a validated `weixin://wxpay/` Native QR.
- A normal mobile browser explains that the order must be opened in WeChat or
  scanned from another device; it does not invent an H5 or app-deep-link flow.
- The WeChat embedded browser uses the published XORPay OpenID redirect before
  creating the JSAPI order. The one-use state is bound to the user, order,
  configuration version, and credential generation, then completed with
  database compare-and-swap protection.
- OpenID is encrypted at rest, excluded from logs and public responses, and
  cleared after the JSAPI start snapshot is committed.
- `WeixinJSBridge` success changes only the interaction state. Final success
  still requires a verified callback or background provider confirmation.

Both themes implement loading, awaiting payment, confirmation, success,
expiry, unavailable, network-recovery, mobile, and desktop states. Persistent
payment state is rendered in the page rather than delegated to transient
toast notifications.

## Merchant payment limits

All enabled Epay, Stripe, XORPay, Creem, Waffo, and Waffo Pancake routes support
merchant-configured:

- single-payment limits;
- daily limits using a configured merchant timezone;
- active-order reservations;
- paid usage;
- release after failure or expiry;
- atomic multi-node capacity checks;
- administrator usage and over-limit diagnostics.

All amounts are stored and compared as integer minor currency units. A policy
is keyed by gateway, provider method, currency, and merchant day. It is not a
global assumption about every provider account.

The optional XORPay presets reflect only the supplied merchant reference
materials: Alipay face-to-face must remain below CNY 2,000 per payment and CNY
20,000 per day; current WeChat reference values are CNY 20,000 per payment and
CNY 50,000 per day. The UI therefore presets CNY 1,999.99 and CNY 19,999.99 for
the strict Alipay bounds. Alipay PC, WAP, and Alipay JSAPI reference limits
remain disabled future-product information because those official Alipay
products are not connected.

## Administrator information architecture

Payment administration is split into:

- payment overview and task health;
- user payment routes;
- provider-specific connection settings;
- device and checkout capabilities;
- merchant limits and current usage;
- orders, callbacks, and manual-review exceptions;
- normal credential rotation;
- advanced and emergency operations;
- legacy Stripe recurring inventory only when historical rows exist.

Each section saves independently. A save owns one updateable notification. If
the write succeeds but the subsequent refresh fails, the result is reported as
"saved, but refresh failed" instead of a false save failure. Opening or
canceling step-up verification does not emit an error notification.

Secrets are never returned after save. Normal rotation retains a bounded prior
credential generation for pre-rotation orders and delayed callbacks. Emergency
revocation requires privileged step-up verification, an impact preview, a
reason, confirmation, and an audit record; affected unfinished orders move to
manual review.

Epay, Stripe, and XORPay expose generation-aware normal rotation and emergency
revocation. Creem, Waffo, and Waffo Pancake currently keep only one active
credential set, so their emergency action is explicitly described as disabling
the current credentials rather than revoking a nonexistent previous
generation. The preview identifies every unfinished or callback-dependent
order that will move to manual review.

Stripe controls state explicitly that local credential clearing or emergency
revocation does not rotate or delete a Stripe API key in the Stripe Dashboard,
expire a Checkout Session, cancel a subscription, or issue a refund. Those are
separate Stripe-side actions. Current purchases use one-time Checkout payment
semantics; legacy recurring inventory remains read-only and conditional.

## Multi-node deployment contract

Single-node deployments retain SQLite, MySQL, and PostgreSQL support, with
Redis either enabled or disabled. Multi-node payment deployment requires all
of the following:

- one shared MySQL or PostgreSQL primary database;
- one shared Redis deployment;
- a unique, stable `NODE_NAME` for every application replica;
- `PAYMENT_MULTI_NODE_ENABLED=true` on every replica, one shared
  `PAYMENT_CLUSTER_ID`, the same comma-separated `PAYMENT_CLUSTER_NODES`
  allowlist, and the same strict-majority
  `PAYMENT_CLUSTER_MIN_LIVE_NODES` value;
- identical session, crypto, payment-encryption, callback, provider, and
  payment configuration on every node;
- synchronized time and externally reachable HTTPS callback routes;
- no payment state stored in local files or process memory.

Every node reports non-secret runtime fingerprints. New payment creation and
provider workers fail closed when live peers disagree on schema version,
configuration version/fingerprint, payment key identity, session key identity,
database type, Redis target, cluster identity, membership policy, or minimum
live count. Unknown or duplicate logical nodes are rejected. The shared Redis
membership witness must exactly match each database's live instance inventory,
so even a wrong-database subset that reaches the configured minimum cannot
serve payment traffic or claim create, reconcile, or maintenance work.
Callback routes can enter any ready node and
settlement remains protected by event deduplication and transactional ledger
fences. When readiness fails, automatic callback processing also fails closed
with a retryable response. XORPay publishes a bounded retry schedule, while
arbitrary Epay-compatible gateways do not share one guaranteed retry contract;
a prolonged pause may require provider reconciliation or administrator review.

The configured node list is an allowlist, not an all-nodes-must-be-online rule.
A strict majority is always required so two disjoint partitions cannot both
serve payment. A three-node allowlist with a minimum of two lets one replica
restart without pausing payment; a two-node cluster necessarily pauses payment
while either member is unavailable.

Maintenance work is also leased through the shared database. No irreplaceable
master node owns creation repair, reconciliation, expiry, or quote cleanup.

### Rolling upgrade rule

Do not create new payments while old and new application binaries coexist.
Keep callback endpoints externally reachable, drain browser creation traffic,
roll every application and worker replica, wait for all runtime fingerprints
to agree, then re-enable payment creation. During an intentional readiness
mismatch, automatic callback handling returns a retryable error and must be
covered by the provider's verified retry behavior or administrator
reconciliation. This prevents a pre-`v0.2.1` synchronous worker from racing a
fenced background create task. Database migrations are additive, but
mixed-version payment creation is intentionally not claimed.

SQLite must never be shared by multiple application nodes.

## Upgrade and rollback boundary

The supported source upgrade for this release candidate is a clean install or
an additive upgrade from `v0.2.0`, after backup and rehearsal against a restored
copy of the installation database. Direct `v0.1.7` through `v0.1.14` upgrades
remain unsupported. A direct `v0.1.6` to `v0.2.1` production upgrade is not
claimed until that exact path is rehearsed separately.

Before migration:

1. Stop new payment creation, but keep callbacks reachable.
2. Record pending orders, manual-review events, active limit reservations,
   callback backlog, and legacy Stripe inventory.
3. Back up the primary database and any separate log database.
4. Restore the backup in isolation and run the migration and payment smoke
   before touching production.
5. Record all shared secrets and provider configuration without writing them
   into logs, source control, or release artifacts.

The additive `top_ups.credit_quota_snapshot` column freezes the exact credit
for new retained Waffo and Waffo Pancake orders. A `v0.2.0` pending order from
either provider has no trustworthy snapshot; when its callback arrives,
`v0.2.1` moves it to manual review instead of guessing from the current
`QuotaPerUnit`. The snapshot is administrator-only data and is never serialized
through the user payment API.

Rollback after new financial writes is a database and provider-reconciliation
operation, not only a binary downgrade. Preserve a post-failure snapshot,
account for provider events received after the backup, and prefer a forward fix
when authoritative payment or ledger writes have already resumed.

## Release acceptance evidence

A candidate tag starts the protected staging workflow. The tag, its run-scoped
images, and its assets must not be described as stable, deployable, or the final
release alias until all blocking repository gates pass from the final commit:

- complete Go test and race suites plus `go vet ./...`;
- Default typecheck, tests, lint, copyright, seven-locale synchronization, and
  production build;
- Classic tests, formatting/lint checks, seven-locale payment checks, and
  production build;
- clean migration and representative additive-upgrade checks for SQLite,
  MySQL 5.7.8+, and PostgreSQL 9.6+;
- database and service integration tests for task lease/fencing, concurrent
  limit reservation, callback deduplication, exactly-once settlement, expiry
  release, and stale-reservation repair;
- a real two-process runtime smoke using shared PostgreSQL and Redis for
  cross-node sessions, synchronized readiness, rolling-member tolerance,
  key/identity mismatch fail-close behavior, isolated-database rejection, and
  recovery, plus Redis-disabled single-node runtime checks;
- native amd64 and arm64 image startup, runtime smoke, manifest, SBOM,
  provenance, and signature gates in the protected release workflow.

Repository tests and protocol fixtures may prove the implementation contract;
they cannot prove a particular merchant account's permissions or external
delivery path.

## Explicitly unverified external capabilities

Unless separately recorded by the operator, this release does not claim that a
real merchant has completed:

- an Epay, Stripe, XORPay, Creem, Waffo, or Waffo Pancake charge;
- XORPay WeChat OpenID authorization or JSAPI bridge payment in production;
- provider callback retry, payout, refund, or dispute delivery through the
  deployment's public reverse proxy;
- Alipay official PC website, WAP, or Alipay JSAPI authorization;
- a WeChat H5 payment product through XORPay;
- reverse-proxy, load-balancer, CDN, or cloud access-log redaction for the
  OpenID callback query.

The application callback handler and access logger redact OpenID and state.
Operators must independently configure every external proxy and observability
layer not to retain those query values.
