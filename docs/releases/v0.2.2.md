# v0.2.2

`v0.2.2` is a payment-interface reliability and accessibility release for the
protected `v0.2` line. It builds on the canonical payment, ledger, callback,
credential, and multi-node contracts published in `v0.2.1`; no payment
protocol or database schema changes are included in this release.

This file describes the source release candidate. The exact `v0.2.2` tag is
deployable only after the protected publish-once workflow verifies every
required test, asset, image, signature, attestation, and runtime gate. Existing
tags, releases, images, and assets remain immutable, and the global container
`latest` alias is not promoted by this release.

## Product result

- The user checkout keeps the configured site name and logo stable during
  initial loading and logo refreshes. Failed logos fall back safely to the site
  name instead of exposing an internal payment provider as the brand.
- Payment result copy is shorter and no longer displays the removed
  balance-confirmation explanation. Server-confirmed callbacks or order queries
  remain the only settlement authority.
- WeChat actions use an accessible high-contrast treatment, order countdowns
  stop after terminal or local-expiry states, and expired pages do not show a
  contradictory polling warning.
- Payment operations retain their administrative information density while
  presenting compact mobile metrics, touch-sized scrollable tabs, and distinct
  loading, empty, unavailable, migration, and retry states.
- Payment gateway settings fail closed when the current configuration cannot
  be loaded. Provider validation stays attached to the relevant field and tab,
  while successful step-up verification remains required for protected writes.
- Classic provider panels render on demand so hidden optional integrations do
  not issue requests or fail the whole page. Payment-specific colors use the
  theme's semantic palette and remain readable in both themes.

## Information architecture and security

Payment Gateway remains the configuration parent. General owns the public
payment catalog and shared routing policy; Epay, XORPay, and Stripe own only
their connection and provider-specific capabilities. Payment Operations stays
an independent operational module. This release does not add a second source
of configuration or expose provider, credential, webhook, task, or ledger
terminology on the user checkout.

Provider creation, amount and currency authority, callback signatures,
merchant binding, event deduplication, exactly-once ledger settlement,
credential generations, manual-review audit, and cluster readiness are
unchanged from `v0.2.1`. A failed or ambiguous creation remains unresolved or
reviewable; it is never promoted to paid and never grants balance or access.

## Compatibility and upgrade

- Clean installations and rehearsed upgrades from `v0.2.1` are in scope.
- Earlier supported installations must use the staged `v0.1.6` to `v0.2.0` to
  `v0.2.1` path before upgrading to `v0.2.2`.
- Direct production upgrades from `v0.1.7` through `v0.1.14` remain unclaimed.
- There is no database migration in this release. SQLite, MySQL, PostgreSQL,
  Redis-enabled, Redis-disabled, and payment multi-node behavior retain the
  `v0.2.1` contract.
- Rollback is application-only when no later release has introduced a schema
  change: stop writes, preserve a database backup and payment evidence, then
  restore the previously verified exact-version image digest and configuration.

## Verification boundary

Repository tests, production frontend builds, browser regression fixtures, and
the protected release workflow verify the implemented contracts. Browser
fixtures cover both frontend themes at desktop and mobile widths, including
light and dark checkout, configuration-load failure, operational empty/error
states, and tab interaction without horizontal overflow.

This release does not claim a real merchant charge, production provider
credential, externally delivered callback, live step-up session behind a
production proxy, refund, dispute, payout, or production multi-node deployment
unless an operator records that validation separately.
