# v0.2.0

`v0.2.0` is the first release on the protected `v0.2` line. It adds the
canonical payment and billing architecture, strengthens Epay and Stripe, and
adds XORPay while preserving the supported `v0.1.6` production contract.
This is the `liulixin-lex/xy-api` fork's `v0.2.0` line based on the fork's
`v0.1.6`; it is not the same tag commit or release artifact as the upstream
`QuantumNous/new-api` project's `v0.2.0`.

> **Supported upgrade boundary:** clean installations and direct upgrades from
> the exact `v0.1.6` release are supported. Direct upgrades from `v0.1.7`
> through `v0.1.14` are not claimed or supported by this release.

## Highlights

- Adds XORPay as a first-class gateway alongside Epay and Stripe.
- Uses server-authoritative quotes, canonical payment orders, durable event
  inboxes, immutable ledger entries, debt records, and exactly-once settlement
  fences.
- Hardens callback identity, amount, currency, merchant/account, environment,
  credential-generation, refund, dispute, and legacy-order verification.
- Adds durable billing reservations and recovery across relay, task, video,
  Midjourney, Realtime, subscription, and wallet charging paths.
- Encrypts saved payment credentials with a dedicated deployment key and adds
  safe key rotation, configuration versioning, privileged-operation step-up,
  audit trails, reconciliation, and manual-review tools.
- Adds payment operations, XORPay QR payment, order tracking, subscription, and
  billing-history flows to both Default and Classic frontends, with the new
  payment copy present across all seven supported frontend locales.

Creem, Waffo Pancake, and Waffo remain outside the canonical gateway migration
scope for this release.

## Compatibility and upgrade boundary

The `v0.2` line starts from tag `v0.1.6`, commit
`2014e1d596a482beb10b1943aa292e1fa991f636`. The `v0.1.6` tag, release assets,
images, and historical source are the preservation baseline and must remain
unchanged. Its Git tag is repository-rule protected, while operators should
retain the recorded Release asset checksums and container manifest digest.

The database migration supports SQLite, MySQL 5.7.8 or newer, and PostgreSQL
9.6 or newer. Migrations run automatically during application startup. The new
canonical payment schema and compatibility columns retain legacy payment and
subscription rows for reconciliation or explicit administrative resolution.

On MySQL, startup checks the current schema character width and InnoDB page,
large-prefix, and file-format capacity before changing any table. New tables
are created explicitly with `ROW_FORMAT=DYNAMIC`. An existing v0.1.6
`subscription_orders` table using COMPACT or REDUNDANT is rebuilt as DYNAMIC
before the new full-length provider indexes are added; stored rows and index
values are retained. This DDL needs a metadata lock and temporary disk space
roughly proportional to that table and its indexes, and it can create replica
lag. Confirm free space, use the maintenance window already required below,
and rehearse the conversion on the restored production copy. A failed
capability check is a hard stop—do not bypass it or replace full indexes with
silent prefix indexes.

Do not treat the channel-routing releases from `v0.1.7` through `v0.1.14` as an
upgrade path to `v0.2.0`. Those releases have a different history and schema
surface. An installation on one of those versions must first be assessed and
migrated through a separately tested plan; replacing its binary or image with
`v0.2.0` is not supported.

The stable deployment container is published as
`ghcr.io/liulixin-lex/xy-api:v0.2.0` and by its content digest. The release
pipeline also creates run-scoped candidate references while it performs native
startup, vulnerability, SBOM/provenance, and signature checks. Those candidate
references are internal publication intermediates, not stable deployment
aliases. The global container `:latest` alias intentionally remains on the
prior release line, because moving it would silently auto-upgrade
v0.1.7-v0.1.14 installations through an unsupported path. New v0.2 deployments
must use the manifest digest anchored in the immutable Release.

The repository `docker-compose.yml` is a production-oriented fail-closed
baseline. It refuses to render without an exact GHCR digest reference and
non-empty database, Redis, session, crypto, and payment secrets; its
`config-check` service additionally rejects secrets shorter than 32 characters
and common placeholders, whitespace, low-diversity values, or reuse across
database, Redis, and payment boundaries before the application or data
services start. `SESSION_SECRET` and `CRYPTO_SECRET` may match only for the
documented v0.1.6 legacy-key transition. A non-empty
`PAYMENT_SECRET_KEY_PREVIOUS` is checked by the same policy and must differ
from the new key. Redis, PostgreSQL, and optional service examples are pinned
to exact image digests. Copy `.env.example` to a private `.env`, fill every
`REQUIRED_FOR_COMPOSE` value, run `docker compose config`, then run
`docker compose run --rm config-check`. Never commit the resulting `.env`.

## Mandatory pre-upgrade preparation

1. Confirm by exact tag plus binary checksum or image digest that the running
   source is `v0.1.6`, not a later tag, a mutable `latest` image, or a locally
   modified build.
2. Record the current image digest or binary checksum, environment variables,
   mounted volumes, database engine and version, Redis configuration, reverse
   proxy configuration, and application node count. Store this inventory with
   the rollback backup; do not commit secrets to the repository.
   Preserve the exact `SESSION_SECRET` and `CRYPTO_SECRET` values during the
   upgrade. If `CRYPTO_SECRET` was unset on `v0.1.6`, the application used
   `SESSION_SECRET` as its Redis encryption key; keep that relationship until a
   separate maintenance window clears only this application's Redis namespace
   and rotates the value consistently across every node.
3. Temporarily stop new payment creation and new billable relay traffic, then
   allow active requests to drain. Keep the public callback routes unchanged
   while draining so already-created provider payments can finish.
4. Review and record pending payment orders, manual-review events, open billing
   reservations, unresolved refunds/disputes, and legacy subscription items.
5. Stop every application node before taking the authoritative backup. Database
   and Redis services may remain running.
6. Back up the primary database and rehearse restoration into an isolated
   database. If `LOG_SQL_DSN` points to a separate audit database, back it up
   under the same retention policy.
7. Run `v0.2.0` against a restored copy of the installation's production data
   before changing the live database. A clean or synthetic migration is not a
   substitute for this installation-specific rehearsal.

## Database backup requirements

Use a private backup directory and retain the backup until the release has
completed its operational observation window. The examples below intentionally
prompt for database credentials or rely on the operator's existing protected
credential mechanism; do not put passwords directly on the command line.
Database backups contain user data, tokens, payment records, and potentially
legacy plaintext gateway credentials from `v0.1.6`. Encrypt backups at rest,
restrict access, and store their encryption key separately.

### SQLite

A SQLite-based container deployment commonly stores the database at
`./data/one-api.db`; the repository v0.2 Compose baseline uses PostgreSQL, so
use the PostgreSQL procedure below for that baseline. For an existing SQLite
deployment, stop all application writers first, then create and verify a
SQLite-native backup:

```bash
set -Eeuo pipefail
source_db=./data/one-api.db
backup_dir=backup/v0.2.0
backup_file="$backup_dir/one-api-v0.1.6.sqlite3"
checksum_file="$backup_file.sha256"
install -d -m 0700 "$backup_dir"
[ ! -e "$backup_file" ] && [ ! -e "$checksum_file" ] || {
  echo 'Refusing to overwrite an existing v0.1.6 SQLite backup.' >&2
  exit 1
}
docker compose stop new-api
if [ ! -f "$source_db" ] || [ ! -s "$source_db" ]; then
  echo "Source SQLite database is missing or empty: $source_db" >&2
  exit 1
fi
sqlite3 "$source_db" 'PRAGMA wal_checkpoint(FULL);'
sqlite3 "$source_db" ".backup '$backup_file'"
[ "$(sqlite3 "$backup_file" 'PRAGMA integrity_check;')" = ok ]
sha256sum "$backup_file" >"$checksum_file"
sha256sum --check --strict "$checksum_file"
```

`PRAGMA integrity_check` must print `ok`. If the installation uses a different
SQLite path, use the actual mounted file rather than assuming the Compose
default.

### MySQL

Use a client compatible with the server and take a logical backup while the
application nodes are stopped:

```bash
set -Eeuo pipefail
: "${MYSQL_HOST:?Set MYSQL_HOST to the source MySQL host}"
: "${MYSQL_USER:?Set MYSQL_USER to the source MySQL user}"
: "${MYSQL_DATABASE:?Set MYSQL_DATABASE to the source application database}"
backup_dir=backup/v0.2.0
backup_file="$backup_dir/new-api-v0.1.6.mysql.sql.gz"
checksum_file="$backup_file.sha256"
install -d -m 0700 "$backup_dir"
[ ! -e "$backup_file" ] && [ ! -e "$checksum_file" ] || {
  echo 'Refusing to overwrite an existing v0.1.6 MySQL backup.' >&2
  exit 1
}
mysqldump \
  --host="$MYSQL_HOST" \
  --user="$MYSQL_USER" \
  --password \
  --single-transaction \
  --quick \
  --routines \
  --triggers \
  --events \
  --hex-blob \
  --set-gtid-purged=OFF \
  "$MYSQL_DATABASE" \
  | gzip -9 >"$backup_file"
gzip -t "$backup_file"
sha256sum "$backup_file" >"$checksum_file"
sha256sum --check --strict "$checksum_file"
```

This dump intentionally omits `CREATE DATABASE` and `USE` statements. Create a
new, empty database on an isolated MySQL instance, verify every restore value,
and name the target database explicitly:

```bash
set -Eeuo pipefail
: "${RESTORE_MYSQL_HOST:?Set the isolated restore MySQL host}"
: "${RESTORE_MYSQL_USER:?Set the isolated restore MySQL user}"
: "${RESTORE_MYSQL_DATABASE:?Set the new isolated restore database}"
if [ "$RESTORE_MYSQL_HOST" = "$MYSQL_HOST" ] && \
   [ "$RESTORE_MYSQL_DATABASE" = "$MYSQL_DATABASE" ]; then
  echo 'Refusing to restore into the source MySQL database.' >&2
  exit 1
fi
gzip -dc "$backup_file" \
  | mysql \
      --host="$RESTORE_MYSQL_HOST" \
      --user="$RESTORE_MYSQL_USER" \
      --password \
      "$RESTORE_MYSQL_DATABASE"
```

Start one `v0.2.0` node against that copy. Never restore the rehearsal dump
into the live application database. Host aliases can refer to the same server,
so the operator must also confirm network identity, not only compare strings.

### PostgreSQL

Use a `pg_dump` client compatible with the server. Supply the password through
`.pgpass`, `PGPASSFILE`, or another protected credential mechanism:

```bash
set -Eeuo pipefail
: "${POSTGRES_HOST:?Set POSTGRES_HOST to the source PostgreSQL host}"
: "${POSTGRES_USER:?Set POSTGRES_USER to the source PostgreSQL user}"
: "${POSTGRES_DATABASE:?Set POSTGRES_DATABASE to the source application database}"
backup_dir=backup/v0.2.0
backup_file="$backup_dir/new-api-v0.1.6.postgres.dump"
checksum_file="$backup_file.sha256"
install -d -m 0700 "$backup_dir"
[ ! -e "$backup_file" ] && [ ! -e "$checksum_file" ] || {
  echo 'Refusing to overwrite an existing v0.1.6 PostgreSQL backup.' >&2
  exit 1
}
pg_dump \
  --host="$POSTGRES_HOST" \
  --username="$POSTGRES_USER" \
  --dbname="$POSTGRES_DATABASE" \
  --format=custom \
  --no-owner \
  --no-acl \
  --file="$backup_file"
pg_restore --list "$backup_file" >/dev/null
sha256sum "$backup_file" >"$checksum_file"
sha256sum --check --strict "$checksum_file"
```

Restore the custom-format dump into a newly created database on an isolated
PostgreSQL instance. Pass that target with `pg_restore --dbname` and verify that
it is not the source database before exercising the upgrade:

```bash
set -Eeuo pipefail
: "${RESTORE_POSTGRES_HOST:?Set the isolated restore PostgreSQL host}"
: "${RESTORE_POSTGRES_USER:?Set the isolated restore PostgreSQL user}"
: "${RESTORE_POSTGRES_DATABASE:?Set the new isolated restore database}"
if [ "$RESTORE_POSTGRES_HOST" = "$POSTGRES_HOST" ] && \
   [ "$RESTORE_POSTGRES_DATABASE" = "$POSTGRES_DATABASE" ]; then
  echo 'Refusing to restore into the source PostgreSQL database.' >&2
  exit 1
fi
createdb \
  --host="$RESTORE_POSTGRES_HOST" \
  --username="$RESTORE_POSTGRES_USER" \
  "$RESTORE_POSTGRES_DATABASE"
pg_restore \
  --host="$RESTORE_POSTGRES_HOST" \
  --username="$RESTORE_POSTGRES_USER" \
  --dbname="$RESTORE_POSTGRES_DATABASE" \
  --exit-on-error \
  --single-transaction \
  "$backup_file"
```

PostgreSQL application nodes serialize schema migration with a database
advisory lock, but the first production start should still use one node so logs
and failures are unambiguous. Host aliases can refer to the same server, so
compare the actual server identity as part of the restore rehearsal.

## Required payment configuration

### `PAYMENT_SECRET_KEY`

Set `PAYMENT_SECRET_KEY` before enabling any payment gateway or saving any
gateway credential. It must be a durable, random value of at least 32
characters and must be identical on every application node. Generate and store
it once in the deployment's secret manager; do not regenerate it during each
container start.

One suitable generation command is:

```bash
openssl rand -base64 48
```

Do not reuse `SESSION_SECRET`, `CRYPTO_SECRET`, a database password, or a
provider credential. Never log, commit, or include this key in a support bundle.
Losing it makes encrypted gateway credentials unavailable.

`PAYMENT_SECRET_KEY_PREVIOUS` is only for an explicit key rotation. During a
rotation, deploy the previous and new keys to every node, verify that stored
credentials and pending payment payloads were re-encrypted, and only then
remove the previous key from every node. Never rotate one node independently.

### HTTPS payment callback origin

Set `CustomCallbackAddress` to the public HTTPS **origin** used by providers,
for example `https://payments.example.com`. It must not include a path, query,
fragment, embedded credential, or non-public development address.

The callback routes are:

- Epay canonical: `/api/payment/epay/notify` (`GET` and `POST`)
- Epay legacy top-up adoption: `/api/user/epay/notify` (`GET` and `POST`)
- Epay legacy subscription adoption: `/api/subscription/epay/notify` (`GET`
  and `POST`)
- Epay legacy subscription browser return: `/api/subscription/epay/return`
  (`GET` and `POST`, display-only and never authoritative for entitlement)
- Stripe: `/api/stripe/webhook`
- XORPay: `/api/xorpay/notify`

The TLS certificate and full certificate chain must be valid for the public
hostname. Plain HTTP is accepted only for loopback development and must not be
used for a production gateway. During the upgrade, keep all four Epay routes
above reachable until every `v0.1.6` pending top-up and subscription order has
reached a terminal state or expired and reconciliation is complete. The
reverse proxy must allow the documented callback methods, must not rewrite
query/form fields or request bodies, and must preserve the raw Stripe body and
`Stripe-Signature` header. Configure `TRUSTED_PROXY_CIDRS` only for the actual
proxy networks; do not trust a public or global CIDR.

The application refuses to change `CustomCallbackAddress` while any gateway
order is active. Because Epay and XORPay freeze `notify_url` when an order is
created, their failed or expired orders also retain the old origin for a
30-day late-settlement recovery window. Keep that origin and its routes
reachable throughout the window, reconcile unresolved orders before switching
it, and treat an older late payment as an explicit manual-reconciliation case.
Stripe Price/catalog settings may be disabled without breaking confirmation of
an existing Session, but its API credential cannot be cleared while durable
Stripe history exists unless the audited emergency-revocation flow is used.

All application nodes must share the same primary database, payment encryption
keys, callback origin, and provider configuration. Payment readiness must show
no missing key, callback, mode, account, credential, currency, or Stripe
verification fingerprint before payment creation is re-enabled.

Normal Epay credential rotation keeps the previous generation available for
30 days plus 3 hours: the two-hour local order lifetime, the full 30-day
late-callback recovery window after expiry, and a one-hour margin. Normal
XORPay rotation uses 30 days plus 25 hours for its accepted maximum 24-hour
provider order lifetime, the same recovery window, and the margin. In both
gateways the old generation is limited to orders created before the rotation
cutoff, and another normal rotation is rejected until the overlap ends. The
audited emergency-revocation path bypasses the overlap and immediately
invalidates the affected generation; use it only for a credential incident and
review every dependent order and event.

Configure the Stripe webhook endpoint API version as
`2026-06-24.dahlia`, matching the pinned `stripe-go` SDK. The v0.1.6
`acacia` financial-event train is explicitly supported for upgrade history.
Validly signed refund or dispute events from other trains, preview trains, or
an invalid/missing API version are retained for manual review with no automatic
economic reversal. One-time Checkout payment events remain independently
authoritative only after the server re-fetches and matches the exact Session.
Refund and dispute events locate their order only through the immutable
`ProviderPaymentKey` (the PaymentIntent for canonical Checkout payments), not
through mutable `trade_no` metadata or a Checkout/order key. Each dispute is
tracked independently by `ProviderResourceKey`, derived from its Stripe
dispute ID, and the order's disputed exposure aggregates all still-open dispute
resources so resolving one cannot erase another.

## Upgrade procedure

1. Complete the inventory, traffic drain, database backup, checksum, and
   isolated restore rehearsal above.
2. Configure `PAYMENT_SECRET_KEY` identically on every node and configure the
   HTTPS callback origin and trusted reverse proxy boundary.
3. Verify the `v0.2.0` release assets or container signature before executing
   them. Do not deploy from `latest`.
4. Keep all existing application nodes stopped and start exactly one `v0.2.0`
   node against the live primary database. Schema migration is automatic.
5. Watch startup logs through database initialization. If migration or payment
   secret initialization reports an error, stop immediately; do not start the
   remaining nodes and do not bypass the failure.
6. Verify `/api/status`, administrator login, payment readiness, outstanding
   manual-review items, and billing reconciliation before allowing traffic.
7. Run the deployment smoke matrix: login, token creation, model listing,
   normal relay, streaming relay, exact quota charging, payment quote/start in
   a sandbox or controlled non-charging mode, callback rejection fixtures,
   refund/dispute fixtures, recharge projection, and subscription entitlement.
   Run the repository runtime helper only against an exclusive disposable or
   restored instance with no concurrent user traffic; its temporary mock
   channel is intentionally high priority and its two relay calls create real
   local quota/log records.
8. Start the remaining nodes one at a time. Confirm that every node reports the
   same configuration version and payment readiness.
9. Re-enable normal traffic and new payment creation only after the acceptance
   matrix is complete. Monitor payment events, reservations, debt freezes,
   database errors, callback failures, and quota deltas during the observation
   window.

For a binary deployment, replace the executable only after checksum
verification and keep the verified `v0.1.6` binary with the rollback bundle.
For containers, pin the deployment to the verified `v0.2.0` manifest digest,
not a mutable tag.

## Release asset checksum verification

The core release must contain these eight files:

- `new-api-v0.2.0`
- `new-api-arm64-v0.2.0`
- `new-api-macos-v0.2.0`
- `new-api-v0.2.0.exe`
- `checksums-linux.txt`
- `checksums-macos.txt`
- `checksums-windows.txt`
- `checksums.txt`

Download the core assets into an otherwise empty directory and run the
repository verifier. The full verifier requires GNU/Linux, Bash 4 or newer,
GNU `find`, and GNU `sha256sum`; run it on a Linux workstation or CI runner:

```bash
install -d -m 0700 release-assets/v0.2.0
gh release download v0.2.0 \
  --repo liulixin-lex/xy-api \
  --dir release-assets/v0.2.0 \
  --pattern 'new-api-v0.2.0' \
  --pattern 'new-api-arm64-v0.2.0' \
  --pattern 'new-api-macos-v0.2.0' \
  --pattern 'new-api-v0.2.0.exe' \
  --pattern 'checksums-linux.txt' \
  --pattern 'checksums-macos.txt' \
  --pattern 'checksums-windows.txt' \
  --pattern 'checksums.txt'
(cd release-assets/v0.2.0 && sha256sum --check --strict checksums.txt)
./scripts/release/verify-release-assets.sh release-assets/v0.2.0 v0.2.0
```

The verifier rejects missing, extra, or symbolic-link platform assets,
validates each platform checksum file, and verifies the aggregate
`checksums.txt` against a freshly computed value. If the aggregate file is not
present in a local build directory, the verifier creates it deterministically.
A checksum proves transfer
integrity against the files published in the same GitHub Release; it is not a
substitute for container signature verification or control of the repository
release channel.

On macOS without GNU coreutils, download `new-api-macos-v0.2.0` and
`checksums-macos.txt` into the same empty directory and run:

```bash
shasum -a 256 -c checksums-macos.txt
```

This verifies the macOS file against the published checksum but does not run
the full eight-file inventory policy. The macOS executable is a universal
binary containing both `x86_64` and `arm64` slices.

The Windows desktop package is staged by the Electron workflow and is part of
the same publication gate. Verify its three assets separately:

```bash
gh release download v0.2.0 \
  --repo liulixin-lex/xy-api \
  --dir release-assets/v0.2.0-desktop \
  --pattern 'New-API-App.0.2.0.exe' \
  --pattern 'New-API-App.Setup.0.2.0.exe' \
  --pattern 'checksums-electron-windows.txt'
(cd release-assets/v0.2.0-desktop && \
  sha256sum --check --strict checksums-electron-windows.txt)
```

The twelfth release asset, `container-images.json`, is generated only after the
multi-architecture image build, native startup smoke, manifest inspection, and
Sigstore verification succeed. GitHub Immutable Releases then anchors its
version tag, source commit, exact image digest(s), and certificate identity
alongside the binary assets.

## Container manifest and signature verification

The authoritative container is
`ghcr.io/liulixin-lex/xy-api:v0.2.0`. The release workflow is configured to
produce amd64 and arm64 images, a multi-architecture manifest, SBOM/provenance
attestations, and keyless Sigstore signatures from the repository's pinned
Docker workflow.

The application source and provenance remain bound to the immutable v0.2.0
tag commit. The container publication orchestration is pinned separately to
protected commit `fcf3d0d669087e443ee39a20e56790a598ffec49`, which fixes the
multi-architecture build parameters without changing the tagged application
source.

Verify the versioned manifest with `cosign` before pulling or deploying it:

```bash
set -Eeuo pipefail
install -d -m 0700 release-assets/v0.2.0-container
gh release download v0.2.0 \
  --repo liulixin-lex/xy-api \
  --dir release-assets/v0.2.0-container \
  --pattern 'container-images.json'
CONTAINER_MANIFEST=release-assets/v0.2.0-container/container-images.json
RELEASE_JSON=$(gh api \
  repos/liulixin-lex/xy-api/releases/tags/v0.2.0)
jq --exit-status \
  '.tag_name == "v0.2.0" and
   .draft == false and
   .prerelease == false and
   .immutable == true' <<<"$RELEASE_JSON" >/dev/null
API_MANIFEST_DIGEST=$(jq --raw-output \
  '.assets[] | select(.name == "container-images.json") | .digest' \
  <<<"$RELEASE_JSON")
LOCAL_MANIFEST_DIGEST="sha256:$(sha256sum "$CONTAINER_MANIFEST" | awk '{print $1}')"
[ "$LOCAL_MANIFEST_DIGEST" = "$API_MANIFEST_DIGEST" ]
jq --exit-status \
  '.schema_version == 1 and
   .release.tag == "v0.2.0" and
   (.release.source_commit | test("^[0-9a-f]{40}$")) and
   (.images.ghcr.digest | test("^sha256:[0-9a-f]{64}$"))' \
  "$CONTAINER_MANIFEST" >/dev/null
IMAGE_TAG=$(jq --raw-output '.images.ghcr.tag' "$CONTAINER_MANIFEST")
IMAGE_DIGEST=$(jq --raw-output '.images.ghcr.digest' "$CONTAINER_MANIFEST")
IMAGE=$(jq --raw-output '.images.ghcr.reference' "$CONTAINER_MANIFEST")
SOURCE_COMMIT=$(jq --raw-output '.release.source_commit' "$CONTAINER_MANIFEST")
CERTIFICATE_IDENTITY=$(jq --raw-output \
  '.cosign.certificate_identity' "$CONTAINER_MANIFEST")
EXPECTED_SOURCE_COMMIT='ea3c0ef6cccddcf252f14f03abb11b2b74366233'
EXPECTED_CERTIFICATE_IDENTITY='https://github.com/liulixin-lex/xy-api/.github/workflows/docker-build.yml@fcf3d0d669087e443ee39a20e56790a598ffec49'
[ "$IMAGE_TAG" = 'ghcr.io/liulixin-lex/xy-api:v0.2.0' ]
[ "$IMAGE" = "ghcr.io/liulixin-lex/xy-api@${IMAGE_DIGEST}" ]
[ "$SOURCE_COMMIT" = "$EXPECTED_SOURCE_COMMIT" ]
[ "$CERTIFICATE_IDENTITY" = "$EXPECTED_CERTIFICATE_IDENTITY" ]
cosign verify \
  --certificate-identity "$CERTIFICATE_IDENTITY" \
  --certificate-oidc-issuer \
    'https://token.actions.githubusercontent.com' \
  "$IMAGE"
docker buildx imagetools inspect "$IMAGE"
./scripts/release/verify-image-index.sh "$IMAGE" v0.2.0 "$EXPECTED_SOURCE_COMMIT"
CURRENT_TAG_DIGEST=$(docker buildx imagetools inspect "$IMAGE_TAG" \
  --format '{{json .Manifest}}' | jq --raw-output '.digest')
[ "$CURRENT_TAG_DIGEST" = "$IMAGE_DIGEST" ]
```

Read the digest from the immutable GitHub Release anchor and use that digest for
every signature, manifest, smoke, and deployment operation. The final equality
check detects a tag change during verification; the anchored digest reference
remains authoritative even if a registry tag is later moved. The signer must be
the exact SHA-pinned reusable Docker workflow recorded above and the issuer
must be GitHub Actions OIDC.
Fulcio uses the reusable job's `job_workflow_ref` as the certificate identity.
Do not broaden the workflow, repository, commit, or issuer identity. The manifest
must list both `linux/amd64` and `linux/arm64`; the verifier also parses the
non-empty SPDX 2.3 SBOM and SLSA v1 provenance for both platforms and requires
every provenance record to bind to the released version and source commit.
Record the verified manifest digest and deploy it as:

```text
ghcr.io/liulixin-lex/xy-api@sha256:<verified-manifest-digest>
```

If a Docker Hub mirror is published, verify it independently rather than
assuming that it has the GHCR digest. Never use `latest` for a controlled
upgrade or rollback.

Run the startup smoke helper on matching amd64 and arm64 runners before the
GitHub Release becomes public:

```bash
./scripts/release/smoke-image.sh "$IMAGE" v0.2.0 amd64
```

Each smoke run checks the pulled architecture, embedded version, container

## Failed release workflow recovery

The published `v0.2.0` Git tag and GitHub Release assets are protected by
GitHub Immutable Releases. Versioned container tags are publish-once by project
policy and workflow enforcement; the content-addressed manifest digest is the
authoritative immutable identity. If a post-tag job fails, use GitHub Actions
**Re-run failed jobs** on the same workflow run so successful build artifacts
and verified candidate digests are reused. Do not move or recreate the tag,
overwrite an asset, delete a versioned image, or start a full rebuild merely to
bypass a failed gate.

If an explicit manual recovery dispatch is required, dispatch the workflow on
the tag ref itself so the OIDC signing identity remains tag-bound. Prefer
**Re-run failed jobs** on the original run because it reuses successful
artifacts and avoids unnecessary builds. A full recovery run can intentionally
produce a different provenance-bearing candidate digest; if the publish-once
version tag already exists, the workflow does not replace it. It resumes only
after the existing digest passes the exact platform, SPDX/SLSA content,
released version/commit, and tag-bound Sigstore identity checks:

```bash
gh workflow run release.yml \
  --ref v0.2.0 \
  -f tag=v0.2.0 \
  -f confirm_tag=v0.2.0
```

Running the same inputs from `main` is intentionally rejected.

If the publication request succeeded but its final confirmation job failed,
the same recovery path accepts only the exact immutable public Release. It
rechecks the tag-bound source, release body, complete asset inventory, rebuilt
core binaries, every checksum and API asset digest, the anchored container
manifest, signatures, and attestations; it never attempts to return the
Release to draft or upload over an immutable asset.

The repository administrator must enable GitHub Immutable Releases, verify the
setting through the repository administration API immediately before creating
the tag, and set the
`IMMUTABLE_RELEASES_VERIFIED_TAG=v0.2.0` Actions variable. The tag-specific
variable is a one-release fail-closed ceremony marker because the standard
workflow token does not have repository-administration read permission; the
next version cannot reuse it. The public release is also required to report
`immutable: true` before the workflow declares success. Draft assets can be
staged and verified, but once the release becomes public its tag and assets
cannot be edited or returned to draft. If the publication API result is
ambiguous, the workflow leaves the versioned images and release state untouched
for explicit verification rather than guessing.

Run and verify the release ceremony immediately before creating the tag:

```bash
set -Eeuo pipefail
repo=liulixin-lex/xy-api
gh api --method PUT "/repos/$repo/immutable-releases"
[ "$(gh api "/repos/$repo/immutable-releases" --jq '.enabled')" = true ]
gh variable set IMMUTABLE_RELEASES_VERIFIED_TAG \
  --repo "$repo" \
  --body v0.2.0
[ "$(gh variable get IMMUTABLE_RELEASES_VERIFIED_TAG --repo "$repo")" = v0.2.0 ]
```

Stable release build artifacts and verified candidate digest records are
retained for 30 days to preserve the same-run recovery path. Investigate and
rerun a failed publication promptly; artifact retention is not permission to
move a tag, overwrite an asset, or replace a version image.

A full workflow rerun can produce a different provenance-bearing candidate
digest even from the same source. If the publish-once `v0.2.0` image tag already
exists, that signed, attested digest remains authoritative and is never moved
to the fresh candidate. An unsigned image, an unexpected signing identity, a
missing or malformed attestation, or provenance bound to another version or
commit fails closed and requires investigation rather than replacement.

## Rollback strategy

Rollback is a database restoration operation, not only a binary or image
downgrade. A `v0.1.6` process must not be pointed at a database that has accepted
new `v0.2.0` payment, ledger, reservation, debt, or entitlement writes and then
be treated as financially authoritative.

1. Block new user and payment writes and stop every application node.
2. Preserve a separate post-failure database snapshot and application logs for
   investigation. Never overwrite the pre-upgrade backup.
3. Record provider payments, refunds, disputes, and callbacks that occurred
   after the pre-upgrade backup. Those external facts are not removed by a
   local database restore.
4. Restore the complete pre-upgrade database backup into a clean database or
   volume. Do not manually drop only the new tables or columns.
5. Clear only this application's Redis database or cache namespace, or point the
   rollback deployment at a clean dedicated Redis database. Never run
   `FLUSHALL` against a Redis service shared by other applications.
6. Restore the recorded `v0.1.6` image digest or verified binary and its original
   configuration. Start one node and verify database integrity, `/api/status`,
   login, relay, and quota behavior before adding nodes.
7. Keep payment creation disabled until every provider event that occurred
   after the backup has been reconciled. Such events may require a provider-side
   refund or explicit manual fulfillment; `v0.1.6` cannot safely infer the new
   canonical accounting history.

When the migration completed and financial writes have already resumed, a
forward fix on `v0.2` is normally safer than restoring stale state. Never
overwrite the preservation-baseline `v0.1.6` or publish-once `v0.2.0` tags,
release assets, or image tags as part of rollback; deploy by recorded digest.

## Known limitations

- No real Epay, Stripe, or XORPay merchant charge is asserted as verified by
  this release. Provider sandbox, protocol fixtures, signature validation, and
  non-charging probes do not prove a specific merchant account's settlement,
  payout, webhook delivery, or refund configuration. Each operator must perform
  its own controlled merchant acceptance test when practical.
- Direct upgrades from `v0.1.7` through `v0.1.14` are not promised. They require
  a separately reviewed migration plan and must not use the `v0.1.6` upgrade
  procedure by assumption.
- A repository test fixture cannot prove an individual installation's data.
  Backup restoration and migration against a copy of production data remain a
  mandatory per-installation gate.
- XORPay's public query API can confirm payment state but cannot recover QR
  content lost with the original create response. The canonical order remains
  recoverable through query, webhook, expiry, or manual review; a duplicate
  upstream order must not be created for the same request.
- Stripe credential and Price probes cannot prove public webhook delivery.
  Callback ingress must be tested from outside the deployment network.
- The in-process protected-fetch client revalidates direct destinations and
  redirects. A configured outbound HTTP proxy or external Worker is a separate
  operator-controlled trust boundary and must enforce equivalent egress and
  destination policy; the application cannot inspect that external hop.
- The compatibility container still runs as root so existing v0.1.6 `/data`,
  SQLite, and log volumes retain their ownership and rollback semantics. Run it
  with the documented narrow volume mounts and container isolation; changing
  to a non-root runtime requires a separately rehearsed volume-ownership
  migration and is not claimed by this release.
- Browser traffic and client code can be inspected. Security relies on TLS,
  server-side pricing and settlement, provider signatures, exact contract
  checks, idempotency, and durable audit evidence—not JavaScript obfuscation or
  hiding payment requests.
- GitHub Release executables are checksum-verified but are not independently
  Sigstore-signed in this release. The Windows standalone binary and Electron
  installers are not Authenticode-signed, so Windows may show an
  unknown-publisher or SmartScreen warning. The keyless Sigstore verification
  described above applies to container images, not to GitHub Release
  executables.
- The universal macOS binary is not signed with an Apple Developer ID and is
  not notarized. Gatekeeper or an organization's endpoint policy may therefore
  block or warn about the downloaded binary even after its checksum is
  verified.
- Default's `zh-TW` locale retains pre-existing English-language UI copy
  outside the new payment flows. The locale gate enforces the exact seven-file
  key set and reports untranslated results only for locales covered by its
  equality-based heuristic; `zh-TW` is explicitly marked as not evaluated
  instead of being reported as a false zero. The new payment copy is present
  in all seven locales, but this release does not claim complete historical
  `zh-TW` translation coverage.
- Classic retains pre-existing translation completeness debt outside the new
  payment flows. The v0.2.0 gate verifies the new payment copy and the Classic
  build/tests, but it does not claim that every historical Classic string is
  translated in every locale.
